# 刷着玩的知识选择题产品需求分析

> [!info] 核心判断
> 这个产品要做的是“内容服务平台”，不是“问答 Agent”。用户不是带着明确问题来问，而是在碎片时间被话题吸引，先做选择，再通过选项理由、群体分布和高价值用户判断获得认知印证与升级。

## 一、定位

### 1.1 产品定位

**一个刷着玩的判断力训练 App：用热点、日常场景和真实决策问题做题卡，让用户先选择，再看到不同选项背后的高价值理由、群体分布和高手判断。**

它不是传统问答产品，也不是纯知识摘要产品：

| 不做什么 | 原因 |
|---|---|
| 不做问答 Agent | 用户必须先有明确问题；一旦答错或答浅，信任损失很大 |
| 不做知乎摘要器 | 只摘要会变成低配 Headway/Blinkist，缺少参与感 |
| 不做考试刷题 App | 本项目的题目多为真实处境判断，不是标准答案训练 |
| 不做纯社区 | 初期内容质量更重要，过早开放社区会稀释信任 |

真正要做的是：

```text
话题吸引 -> 选择参与 -> 平台答案揭示 -> 用户评论区 -> 分布印证 -> 保存/关注/分享
```

### 1.2 用户分析：给谁用

本项目不优先按性别、年龄、城市分层，而按**认知水平与决策成熟度**分层。因为同样是 25 岁用户，有的人只想看热闹，有的人已经需要职业/消费/关系/学习上的判断印证。

#### 核心用户画像

核心用户包括两类：一类是中等学历、生活中打拼的中流阶层或准中流阶层；另一类是高等学历人群。高等学历用户也有同样的部分需求：想在碎片时间补经验、看他人判断、验证自己的选择，同时也希望自己的选项、评论和主页记录被别人看到，用来营造“我会判断、我有观点”的人设。

典型状态是：

- 知道自己需要提升认知，但读书/系统课程太重；
- 刷短视频有娱乐感，但刷完空；
- 看知乎/小红书/公众号觉得信息多，但筛不出重点；
- 遇到职业、消费、学习、人际、城市选择时，不知道该优先看什么变量；
- 希望用低成本方式知道“多数人怎么想”和“更有经验的人怎么看”。
- 希望自己的选择和评论在主页中形成可被他人理解的人设资产。

#### 用户认知分层

| 用户层级 | 用户状态 | 核心痛点 | 产品提供的价值 |
|---|---|---|---|
| L0 小白用户 | 没框架，不知道怎么判断 | 不知道问题该从哪里想 | 用题干和选项降低理解门槛 |
| L1 印证型用户 | 有直觉，但不确定 | 想知道自己判断是否离谱 | 展示全部用户选项分布 |
| L2 提升型用户 | 有基本认知，想变强 | 想知道高手和普通人的差异 | 展示高价值用户/超高价值用户分布 |
| L3 表达型用户 | 有经验，愿意输出 | 不想写长文，但愿意说明为什么选 | 选择评论维度，必要时补一句短评论 |
| L4 高价值用户 | 能持续输出高质量判断 | 希望自己的判断被更高权重地看见 | 进入高价值用户分布样本，而不是决定标准答案 |

这套分层的重点是：**用户既是消费者，也是潜在内容贡献者。** 初期用户主要消费，后期用户的选择、理由、保存、分享会形成数据飞轮。

### 1.3 使用场景

| 场景 | 用户心理 | 典型题目 |
|---|---|---|
| 通勤/睡前刷一刷 | 想刷点比短视频有营养的东西 | “年轻人第一份工作最该看什么？” |
| 遇到现实选择 | 想验证自己的判断 | “买课前最该先看什么？” |
| 职业焦虑 | 想补经验和坑点 | “大厂边缘岗和小厂核心岗怎么选？” |
| 学习产品/商业/AI | 想用轻方式建立框架 | “AI 产品经理最容易被高估的能力是什么？” |
| 社交分享 | 想表达立场，看看别人怎么选 | “你会选稳定高薪还是高风险增长？” |

### 1.4 交付方式

建议交付顺序：

1. **第一阶段：H5 / Web 原型**
   - 用于验证题卡、作答、揭示、分布展示、保存/分享。
   - 不需要一开始上原生 App。
   - 主页面采用类似知乎/小红书的信息流呈现：用户先浏览问题卡片，再进入问题详情看选项、平台答案和评论区；不是抖音式全屏沉浸刷题。

2. **第二阶段：移动端 App 或小程序**
   - 如果验证出高频打开和分享，再做更完整的客户端。
   - 小程序适合传播，App 适合长期留存。

3. **后台 CMS**
   - 必须尽早做轻量后台，因为内容生产是核心。
   - 初期可以比用户端更简陋，但必须能录入题目、选项、解释、来源、审核状态、标签和发布状态。

## 二、功能：从价值视角拆解

### 2.1 功能主线

产品不是先堆注册、搜索、评论，而是先完成一个闭环：

```text
用户被题目吸引
  -> 做出选择
  -> 看到自己选项的理由
  -> 看到全部选项的分布
  -> 看到普通用户/高价值用户/超高价值用户怎么选
  -> 理解这题真正考的变量
  -> 保存、分享或选择评论维度后补一句短评论
```

### 2.2 大功能一：题卡质量 SOP 与编辑标准

**对应痛点**：这个项目最难规模化的不是开发，而是题卡质量。平台答案如果像 AI 套话，产品立刻死亡。用户不是来读正确废话的，用户是来找“原来还能这么想”的。

产品价值：

- 先定义什么问题值得做成题卡，而不是先追求题库数量；
- 区分真实处境判断题和低价值冷知识，避免主 Feed 变成冷知识 App；
- 规定选项如何互斥、如何制造真实判断冲突；
- 规定每个选项理由必须包含变量、边界、反例和案例；
- 定义什么叫“高价值解释”，什么样的解释算 AI 套话；
- 明确评论区哪些内容只进评论，哪些未来可能反哺平台答案。

规模化前必须先定义题卡 SOP：

| SOP 项 | 要回答的问题 |
|---|---|
| 入选标准 | 什么问题值得做成题卡？它是否来自真实处境、真实取舍、真实信息差？ |
| 排除标准 | 什么题目只是冷知识、百科题、考试题，不能进主 Feed？ |
| 选项标准 | 选项是否互斥？是否都有人会真实选择？是否避免明显正确/明显错误？ |
| 理由标准 | 每个选项的理由是否包含变量、边界、反例、案例或来源？ |
| 高价值解释标准 | 用户看完是否会产生“原来还能这么想”的感觉？ |
| AI 套话识别 | 是否只是正确废话、空泛建议、没有变量、没有边界、没有反例？ |
| 评论边界 | 哪些用户评论只进入评论区，哪些只作为后续编辑参考，不直接改写平台答案？ |

现阶段不该先工程化 AI 生产 pipeline。应该先人工做 30-100 张题卡，把题卡质量标准打出来。

一句话：**这个项目不是缺 AI 生产，而是缺一套能判断 AI 产物好坏的编辑标准。**

MVP 形态：

- 30-100 张人工打磨题卡；
- 每张题卡都有题干、选项、选项理由、边界、反例和来源；
- 建立一份题卡编辑 checklist；
- 每张题卡上线前先过“冷知识排除”和“AI 套话排除”；
- 先用人工标准验证内容质量，再考虑 AI 辅助生产和 CMS 自动化。

### 2.3 大功能二：话题题卡 Feed

**对应痛点**：用户想学东西，但没有明确问题，也不愿意读长内容。

产品价值：

- 用热点、职业、消费、AI、商业、日常决策等话题吸引用户；
- 每张卡只解决一个判断问题；
- 用选择题降低参与门槛；
- 让用户不需要主动搜索，也能进入知识消费。

MVP 形态：

- 单张题卡；
- 2-4 个选项；
- 上下滑或点击下一题；
- 领域标签：职业、AI、消费、商业、生活决策。

不建议第一版做复杂推荐。初期用人工精选专题比算法更可靠。

### 2.4 大功能三：先选择后揭示

**对应痛点**：传统知识内容是被动阅读，用户容易看完就忘。

产品价值：

- 选择动作让用户先暴露自己的判断；
- 选完再看解释，会产生“我为什么这么想”的反思；
- 解释不只给正确答案，而是说明每个选项的成立条件和盲区。

每个选项下应包含：

- 为什么有人会选；
- 它成立的条件；
- 它忽略的变量；
- 典型案例/来源依据；
- 一句话认知升级。

### 2.5 大功能四：选项分布与认知印证

**对应痛点**：用户有判断，但需要知道多数人怎么选，来验证自己是否偏离常识。

产品价值：

- 作答后以条形图展示 A/B/C/D 的整体选择分布；
- 用户可以看到自己和大众是否一致；
- 分布本身会制造继续看的动机。

基础展示：

```text
A 大厂边缘岗        ████████████ 42%
B 小公司核心岗      ████████ 29%
C 继续读研          █████ 18%
D 先 gap 探索       ███ 11%
```

这不是简单投票，而是“认知对照”。用户会自然追问：为什么多数人选 A？为什么我选 B？为什么高手可能不这么选？

### 2.6 大功能五：高价值用户分层与分层分布

**对应痛点**：用户不只想看多数人怎么选，还想知道“更有经验的人怎么选”。

这是你新增的关键功能点。推荐设计为：

| 层级 | 进入方式 | 在产品中的作用 |
|---|---|---|
| 普通用户 | 注册/匿名作答 | 提供大众选择分布 |
| 高价值用户 | 在某领域的公开短评论长期获得保存、有用反馈、同领域用户认可，且低质率低 | 提供更高信号的选择分布 |
| 超高价值用户 | 在某领域持续产生高信号判断，且选择和理由被大量用户参考 | 形成领域判断样本和分布权重 |

展示方式：

```text
A 大厂边缘岗 42%
  普通用户 48% / 高价值用户 31% / 超高价值用户 21%

B 小公司核心岗 29%
  普通用户 25% / 高价值用户 41% / 超高价值用户 34%
```

这里的产品价值很强：

- 小白看整体分布，获得常识印证；
- L1/L2 用户看高价值用户分布，获得认知差距；
- L3/L4 用户有表达动机，但不需要等待平台采纳；
- 平台获得结构化理由信号，而不是把每条评论都变成待审核内容。

参照机制上，Stack Overflow 把 reputation 定义为社区信任的粗略度量，声望越高权限越多；Reddit 的 karma 来自帖子和评论获得的 upvote/downvote，并会影响一些社区的发帖门槛。这说明“贡献质量 -> 声望/权限/可见性”是成熟社区机制，但本项目不能简单照抄点赞等级。更适合把“高价值”绑定到**公开短评论的长期有用反馈、保存率、低质率、领域稳定性**，而不是平台编辑的主观采纳。

### 2.7 大功能六：开放评论区，但平台答案与用户评论分层

**对应痛点**：用户选完后可能想表达“我为什么这么选”，而且确实需要开放评论区；但平台提供的答案和用户评论前期必须完全分开，不能混成一套“答案”。

这里要区分两层：

| 层级 | 内容来源 | 产品角色 | 前期是否融合 |
|---|---|---|---|
| 平台答案 | 平台/编辑/AI 辅助整理 | 对选项进行结构化解释，承担“答案层” | 不融合 |
| 用户评论 | 用户在题下公开评论 | 表达个人判断、补充案例、提出不同维度 | 不融合 |

前期原则：**平台答案是平台答案，用户评论是用户评论。** 用户评论可以被点赞、排序、沉淀到个人主页，也可以作为后续内容选题参考，但不自动改写平台答案，不进入平台答案解释区。

用户作答后，如果想评论，先不要直接给文本框，而是先让用户选择评论维度：

```text
我选了 A，接下来我要：
[我和答案思考维度一致，我来进行补充]
[虽然我选了这个答案，但我和答案思考维度完全不同]
```

这一步很关键。它把评论分成两类，避免平台事后猜用户意图：

| 评论维度 | 含义 | 产品价值 |
|---|---|---|
| 同维度补充 | 用户认可当前答案解释的思考框架，只是补案例、细节、边界 | 增强该选项下已有解释 |
| 不同维度 | 用户选了同一个选项，但理由不是平台给出的那套思路 | 发现新的判断变量 |

选完评论维度后，直接给一个可选短输入：

```text
写一条评论
```

这样链路保持短：用户只多做一次维度选择，然后直接评论。维度选择不是为了限制评论，而是为了让评论区天然分层，避免平台事后猜用户意图。

公开规则也要清楚：

| 输入类型 | 是否立即可见 | 产品意义 |
|---|---|---|
| 选择选项 | 聚合分布立即可见；个人身份不立即可见 | 进入选项分布，但不自动进入个人主页 |
| 左滑保存至个人画像 | 对主页访客可见 | 该选择进入个人主页的选择记录 |
| 评论维度 | 是 | 区分“同维度补充”与“同选项不同思考维度” |
| 用户评论 | 是 | 进入该题评论区，不进入平台答案 |
| 私有笔记 | 仅自己可见 | 个人复盘，不承诺影响他人 |

评论区排序可以独立于平台答案：

| 排序 | 说明 |
|---|---|
| 综合活性 | 默认排序，综合点赞、评论时间、回复互动、同维度/不同维度覆盖、高价值用户权重，避免评论区只剩旧高赞 |
| 点赞最多 | 看高互动评论 |
| 最新评论 | 看最新讨论 |
| 同维度补充 | 只看补充平台答案维度的评论 |
| 不同维度 | 只看同选项但新思路的评论 |

如果用户的回答不能被别人看到，就不要叫“评论”，只能叫“私有笔记”。评论的最低成立条件是：**用户写完后，至少能在该题评论区看到自己。**

调整后的数据飞轮是：

```text
平台自产题卡
  -> 用户选择
  -> 展示平台答案
  -> 用户选择评论维度
  -> 用户公开评论
  -> 形成各选项下的同维度/不同维度占比
  -> 评论被点赞/回复/保存
  -> 用户分层权重逐渐形成
  -> 高价值用户分布更有参考价值
```

这套机制不要求平台逐条采纳，也不要求用户评论影响平台答案。平台只需要做四件事：反垃圾、低质折叠、统计不同维度占比、维护评论排序。

### 2.8 大功能七：保存、分享与判断资产

**对应痛点**：用户想“学到东西”，但刷完容易遗忘。

产品价值：

- 直接提供私密保存按钮，默认保存到私密收藏；
- 左滑把本题选择保存至个人画像；
- 直接提供分享按钮，默认以图文方式分享“问题 + 选项分布 + 选项答案”；
- 关注该问题，等待后续高质量回答或爆款评论更新；
- 长期生成“我的判断偏好”。

这里要区分两种保存：

| 动作 | 默认可见性 | 产品含义 |
|---|---|---|
| 私密保存按钮 | 私密 | 用户只是想留着以后看 |
| 左滑保存至个人画像 | 公开/半公开 | 用户愿意把这个选择挂到主页，形成“这个人怎么想”的记录 |

推荐交互：

```text
右滑：退出当前问题和选项详情，回到主信息流
左滑：保存至个人画像
按钮：私密保存 / 分享 / 关注问题
```

分享按钮默认产出图文卡片，包含：

- 问题标题；
- 用户选择；
- 选项分布；
- 对应选项的平台答案/解释；
- 可选水印和回流入口。

链接分享、动态链接卡片等形式可以后续探索，不进入 MVP 主链路。

别人打开用户主页时，可以看到一个“选择记录”模块。这里不只展示选择，如果用户对某题写过公开短评论，也一起展示出来：

```text
选择记录
[选择] [评论]

题目：第一份工作更应该优先选什么？
TA 选择：B 小公司核心岗
评论维度：同选项但不同思考维度
TA 评论：我会优先看这份工作能不能让我接触真实业务，而不是只看公司名。
保存人数：2,381
点赞：128
```

排序规则：

| 内容 | 排序 | 说明 | 默认 |
|---|---|---|---|
| 选择 | 话题热度 | 按题卡热度、主页访问者点击/停留、保存互动综合排序 | 默认 |
| 选择 | 选择时间新到旧 | 按用户保存/选择该题的时间倒序 | 可切换 |
| 选择 | 选择时间旧到新 | 按用户保存/选择该题的时间正序 | 可切换 |
| 评论 | 综合活性 | 综合点赞、时间、互动、维度覆盖，保持评论区活性 | 评论默认 |
| 评论 | 点赞最多 | 按该用户评论获得的点赞/有用反馈排序 | 可切换 |
| 评论 | 评论时间新到旧 | 按评论发布时间倒序 | 可切换 |
| 评论 | 评论时间旧到新 | 按评论发布时间正序 | 可切换 |

默认展示“选择”页，并按话题热度排序，因为别人进主页时最想快速理解这个人的代表性判断，而不是看流水账。评论页默认按综合活性排序，避免只看旧高赞导致评论区变死；点赞最多和评论时间顺序作为可切换项。

后期可以生成：

- 你最常选的风险偏好；
- 你经常忽略的变量；
- 你在职业/消费/商业题中的判断倾向；
- 与高价值用户的选择差距。

这是未来高价值转化点。MVP 里建议至少做“私密保存按钮 + 分享按钮 + 左滑保存至个人画像 + 主页选择记录（含评论）”。

### 2.9 大功能八：关注问题、关注流与特别关注

**对应痛点**：用户看到一个好问题后，不一定只想当场看完。他可能想持续追踪这个问题后续有没有新爆款回答、新高赞评论，或者想看自己关注的人最近在选什么、答什么、发什么。

因此产品需要双页面结构：

| 页面 | 主目的 | 内容 |
|---|---|---|
| 主页面 | 探索好问题 | 看好问题、选项、平台答案、评论区 |
| 副页面 | 看关注更新 | 关注的人、关注的问题、特别关注人的最新动态 |

问题关注机制：

```text
用户点击“关注问题”
  -> 该问题进入关注列表
  -> 用户选择是否展示到个人主页
  -> 后续有新爆款回答/高赞评论/平台答案更新
  -> 系统把更新推荐到用户 Feed 或关注副页面
```

关注问题时增加一个可见性开关：

| 动作 | 默认可见性 | 产品含义 |
|---|---|---|
| 关注问题 | 私密或由用户选择 | 只用于后续接收更新 |
| 展示到个人主页 | 用户主动开启 | 表达“我长期关注这个问题”，补充个人画像 |

关注副页面内容：

- 关注问题的新回答、新爆款评论、平台答案更新；
- 关注人的新发帖、新评论；
- 关注人的高价值回答；
- 特别关注人的选项选择。

特别关注机制：

| 关系 | 普通关注 | 特别关注 |
|---|---|---|
| 新发帖 | 进入关注副页面 | 更高权重展示 |
| 新评论 | 进入关注副页面 | 更高权重展示 |
| 选项选择 | 默认不展示或低频展示 | 可以展示到关注副页面 |

特别关注的价值是：用户不仅想看这个人发了什么，也想看这个人在好问题上**怎么选**。这会把“选择”从个人画像资产进一步变成关系流内容。

MVP 可以先做：

- 关注问题；
- 关注用户；
- 特别关注用户；
- 关注副页面；
- 关注问题有高互动新回答/评论时回流到用户 Feed；
- 特别关注人的选项选择进入关注副页面。

### 2.10 大功能九：主题讨论与用户发帖

**对应痛点**：用户不一定每次表达都是为了回答某道选择题。有时只是想记录一个经验、吐槽一个现象、分享一个判断，或围绕某个主题展开讨论。

因此可以在选项和答案主页面之外，加一个**主题讨论 section**。它是副功能，不抢主 Feed 的位置，但能补足三件事：

- 让用户有日常发帖入口；
- 充实用户主页里的个人画像；
- 为平台沉淀非题卡形态的内容资产。

推荐机制：

```text
用户发帖
  -> 系统做简单主题识别
  -> 默认归入一个主题
  -> 用户可手动切换主题
  -> 用户也可选择“不进入主题”
  -> 帖子展示在主题讨论区，同时进入个人主页
```

预设主题不宜太多，先控制在 5-8 个：

| 主题 | 适合内容 |
|---|---|
| 职业选择 | 求职、实习、转行、职场判断 |
| AI 产品 | AI 工具、产品设计、模型体验 |
| 消费避坑 | 课程、工具、租房、购买决策 |
| 学习成长 | 方法论、认知升级、复盘 |
| 城市生活 | 城市选择、生活成本、通勤、租房 |
| 关系与沟通 | 人际、亲密关系、合作、表达 |

发帖不是评论，也不要求绑定某道题。它和主流程的关系是：

| 模块 | 主目的 | 是否核心 |
|---|---|---|
| 题卡选择 | 让用户刷、选、看分布、学判断 | 核心 |
| 题下短评论 | 补充某道题下的公开评论 | 辅助 |
| 主题发帖 | 让用户自由表达，沉淀主页和平台资产 | 副功能 |

用户主页也要展示发帖记录：

```text
主页
  - 选择记录
  - 评论记录
  - 发帖记录
    - 默认按互动热度
    - 可按发布时间新到旧
    - 可按发布时间旧到新
```

MVP 里可以先做轻量发帖，不做完整社区：

- 支持发一段文字；
- 支持系统默认识别主题；
- 支持用户改主题；
- 支持“不进入主题”；
- 支持帖子进入个人主页；
- 支持点赞/有用；
- 暂不做私信、粉丝体系、复杂话题运营。

### 2.11 大功能十：内容生产与审核后台

**对应痛点**：内容为王，初期没有高质量题卡就没有产品。

后台必须支持：

- 录入题干、选项、解释、来源；
- 标注题型、领域、难度、争议度；
- AI 辅助从长内容生成题卡草稿；
- 人工审核和改写；
- 维护评论维度配置；
- 处理公开短评论的折叠、置顶和低质内容；
- 管理主题列表、主题识别规则和用户发帖；
- 管理关注问题更新、关注流推荐和特别关注权重；
- 查看单卡数据：作答率、完读率、保存率、分享率、分布差异。

初期不要追求全自动。先用团队自产内容验证需求，类似内容平台早期先用人工内容冷启动。后台可以支持 AI 草稿，但 AI 只能辅助生产，不能替代题卡质量 SOP 和人工编辑判断。

### 2.12 盈利模式初步方向

按你的方向，初期不收用户费用是合理的。更适合：

| 模式 | 阶段 | 判断 |
|---|---|---|
| 广告 | 中后期 | 可以做，但必须谨慎，不要破坏信任 |
| 导流 | 中后期 | 适合课程、书、工具、咨询，但要和题卡内容强相关 |
| 品牌专题 | 中后期 | 可做“职业避坑专题”“AI 工具专题”，但要标注商业合作 |
| 会员 | 后排 | 可能与“不收用户费用”冲突，暂不作为主线 |
| 决策报告 | 后期 | 基于用户选择生成个性化判断报告，价值更高 |

短期目标不是盈利，而是验证：用户是否愿意刷、是否保存、是否贡献、是否分享。

## 三、分析：优先级排序

### 3.1 用户 Kano 模型

| 用户类型 | 重要性 | 需求性质 | 是否核心 |
|---|---:|---|---|
| L0 小白用户 | 高 | 必须满足：题目好懂、解释清楚 | 核心 |
| L1 印证型用户 | 最高 | 必须满足：看整体选择分布 | 核心 |
| L2 提升型用户 | 最高 | 兴奋型需求：看高价值用户怎么选 | 核心 |
| L3 表达型用户 | 中 | 有了更好：低成本表达理由，不等待采纳 | 次核心 |
| L4 高价值用户 | 中 | 有了更好：身份、影响力、权重 | 次核心 |
| 泛娱乐用户 | 中 | 只想看热闹 | 边缘 |
| 明确求答案用户 | 低 | 更适合问答 Agent/搜索 | 非核心 |
| 系统学习用户 | 低 | 更适合课程/书/训练营 | 非核心 |

核心用户不是“最专业的人”，而是**会在选择后产生印证需求和认知差距的人**。

### 3.2 功能 Kano 模型

| 功能 | Kano 分类 | 原因 | 决策 |
|---|---|---|---|
| 题卡质量 SOP 与编辑标准 | 必须有 | 没有题卡质量标准，平台答案会变成 AI 套话，产品信任直接崩掉 | 第一优先级，先人工验证 |
| 题卡 Feed | 必须有 | 没有 feed 就没有刷的入口 | 原型验证 |
| 先选择后揭示 | 必须有 | 产品差异化核心 | 原型验证 |
| 选项级理由 | 必须有 | 没有理由就只是投票/刷题 | 原型验证 |
| 整体选项分布 | 必须有 | 满足印证需求 | 原型验证 |
| 高价值用户分布 | 兴奋型/差异化 | 形成“高手怎么看”的认知价值 | 原型验证，但可先用模拟数据 |
| 评论维度选择 | 有了更好 | 用户先标注同维度补充/不同思考维度，降低意图识别复杂度 | 原型验证 |
| 公开短评论 | 有了更好 | 允许表达，但不拉长主链路 | 做轻量版 |
| 高价值用户等级 | 有了更好 | 需要数据积累，不靠人工采纳 | 先出方案，MVP 用示意 |
| 私密保存按钮 | 必须有 | 用户需要默认私密保存，不把所有保存都公开到主页 | 原型验证 |
| 分享按钮 | 必须有 | 默认图文分享问题、选项分布和选项答案 | 原型验证 |
| 左滑保存至个人画像 | 兴奋型/差异化 | 把选择沉淀成可被别人查看的判断资产 | 原型验证 |
| 主页选择记录（含评论） | 兴奋型/差异化 | 让用户的选择和公开评论形成社交身份与可信度 | 原型验证 |
| 选择/评论排序 | 有了更好 | 选择按话题热度/选择时间排；评论默认综合活性，支持点赞/时间/维度筛选 | 原型验证 |
| 关注问题 | 有了更好 | 让用户持续追踪好问题的新回答和爆款评论 | 原型验证 |
| 关注问题主页展示开关 | 有了更好 | 用户可决定关注的问题是否进入个人主页画像 | 原型验证 |
| 关注副页面 | 有了更好 | 承接关注的人/问题的最新回答、评论、发帖 | 原型验证 |
| 特别关注 | 兴奋型/差异化 | 特别关注人的选项选择可进入关注副页面 | 原型验证 |
| 主题讨论 section | 有了更好 | 允许用户不围绕题目发帖，补充主页画像和平台资产 | 做轻量版 |
| 主题自动识别 | 有了更好 | 降低发帖分类成本，但用户可选择不进入主题 | 做轻量版 |
| 私密收藏 | 有了更好 | 验证获得感，但不如个人画像关键 | 做轻量版 |
| 分享 | 有了更好 | 验证传播 | 原型验证 |
| 个性化推荐 | 可后排 | 初期数据不足 | 后排 |
| AI 问答追问 | 可后排 | 会把产品拉回问答 Agent | 后排 |
| 复杂评论楼中楼 | 可后排 | 早期容易带来治理复杂度 | 后排 |
| 支付/会员 | 可后排 | 当前方向不收用户费 | 后排 |
| 广告系统 | 可后排 | 未验证留存前不做 | 只出方案 |

### 3.3 原型验证范围

#### 必须做进原型

- 题卡质量 SOP 与编辑 checklist；
- 冷知识排除标准；
- AI 套话排除标准；
- 题卡 Feed；
- 选择题作答；
- 答题后 Reveal；
- 每个选项的高价值解释；
- 整体选项分布条形图；
- 高价值用户/超高价值用户分布展示；
- 评论前选择维度：同维度补充/同选项不同思考维度；
- 每个选项下的同维度/不同维度占比；
- 左滑保存至个人画像；
- 用户主页选择记录，包含公开短评论；
- 选择排序：话题热度默认，支持选择时间新旧排序；
- 评论排序：综合活性默认，支持点赞最多、评论时间新旧、同维度/不同维度筛选；
- 私密保存按钮；
- 分享按钮：默认图文分享问题、选项分布和选项答案；
- 关注问题按钮，并可选择是否展示到个人主页；
- 右滑退出当前问题和选项详情，回到主信息流；
- 关注副页面：关注的人/问题的最新回答、评论、发帖；
- 特别关注：特别关注人的选项选择进入关注副页面；
- 主题讨论 section；
- 用户发帖：默认识别主题，可改主题，可选择不进入主题；
- 用户主页发帖记录；
- 分享；
- 30-100 张人工打磨的真实高质量题卡。

#### 可以用示意图/假数据

- 高价值用户真实等级；
- 超高价值用户样本；
- 用户画像报告；
- 公开短评论流；
- 主题推荐/主题热度榜；
- 后台数据大盘；
- AI 自动生成流程。

#### 只出方案，不进第一版

- 完整 UGC 审核流；
- 复杂社区；
- 开放长评论区；
- 个性化推荐；
- 广告导流系统；
- 领域认证体系；
- 用户成长任务体系。

#### 直接后排

- 问答 Agent；
- 支付会员；
- 私信/粉丝体系/复杂关系运营；
- 长文发布；
- 直播/课程；
- 完整 App 原生开发。

### 3.5 核心风险与应对

| 风险 | 表现 | 应对 |
|---|---|---|
| 内容变低质猎奇 | 用户刷但不保存 | 首发聚焦职业/AI/消费决策，不做泛冷知识 |
| 分布误导用户 | 小样本被当成结论 | 样本不足不展示细分；显示“仅供参考” |
| 高价值用户变成等级崇拜 | 用户盲从高手 | 必须展示理由，不只展示“高手选什么” |
| 开放评论复杂度过高 | 每条评论都要意图识别、排序、审核 | MVP 不做长评论，先做评论维度和可选短评论 |
| UGC 污染内容 | 低质回答变多 | 短评论只进补充流，不直接进入题卡解释区 |
| 过早做 Agent | 用户拿来问具体问题，答错即流失 | MVP 坚持内容服务平台，不做开放问答 |
| AI 味太重 | 解释套话、空泛 | 每张卡必须有人改写，有来源和案例 |

## 四、保障条件：投入评估

### 4.1 人员

如果当前只有两个人，可以做**验证型原型**，但不建议直接做完整 App。

最低配置：

| 角色 | 人数 | 职责 |
|---|---:|---|
| 产品/内容负责人 | 1 | 题型设计、用户访谈、内容标准、需求定义 |
| 内容编辑/研究 | 1 | 生产题卡、改写解释、维护评论维度配置 |
| 全栈开发 | 0.5-1 | H5 原型、数据库、后台、数据埋点 |
| 设计 | 0.2 | 题卡 UI、分布图、分享图 |

两人足够做：

- Figma 原型；
- 手工题卡；
- Notion/Airtable 表格后台；
- 简单 H5 demo；
- 小范围用户测试。

两人不够长期做：

- 稳定客户端；
- 完整社区治理；
- 推荐算法；
- 大规模内容生产；
- 广告/导流商业系统。

### 4.2 时间评估

| 阶段 | 周期 | 目标 |
|---|---:|---|
| 需求与样题 | 1 周 | 明确题型、用户分层、做 30 张样题 |
| 可点击原型 | 1 周 | Figma/H5 验证选择、Reveal、分布展示 |
| MVP 开发 | 3-5 周 | Feed、作答、Reveal、收藏、分享、后台录入 |
| 种子测试 | 2 周 | 100 个用户，验证作答/保存/分享/D1 |
| 评论维度与等级试验 | 3-4 周 | 评论维度、公开短评论、分层分布 |

现实估计：**6-8 周可以做出可验证 MVP；10-12 周可以做出带评论维度、公开短评论和高价值用户分层示意的版本。**

### 4.3 外部依赖

| 依赖 | 用途 | 阶段 |
|---|---|---|
| 内容来源 | 题卡素材、观点、案例 | 从第一天开始 |
| AI API | 长文蒸馏、选项生成、解释草稿 | MVP 可用 |
| 数据库/后台 | 题库、选项、解释、用户选择 | MVP 必须 |
| 数据埋点 | 作答率、保存率、分享率、留存 | MVP 必须 |
| 登录系统 | 区分用户、等级、评论记录 | 第一版可轻量 |
| 分享图生成 | 社交传播 | MVP 建议 |
| 广告/导流接口 | 商业化 | 后排 |
| 小程序/App Store 账号 | 分发 | H5 验证后再做 |

### 4.4 经费评估

探索期不需要大投入，但要把内容和开发成本算进去。

| 项目 | 粗略成本 |
|---|---:|
| AI API / 工具 | 300-1500 元/月 |
| 数据库/托管/对象存储 | 100-800 元/月 |
| 埋点/分析工具 | 0-1000 元/月 |
| 设计工具/协作工具 | 0-500 元/月 |
| 外包 H5 MVP | 1-5 万元一次性 |
| 兼职内容编辑 | 3000-10000 元/月 |

如果两个人自己做原型，最低成本可以控制在每月几百到两千元。真正的成本不在服务器，而在**高质量题卡生产和迭代速度**。

## 六、推荐结论

第一版不要做“大而全知识平台”，只做一个强闭环：

**题卡吸引用户作答 -> 用户看到平台答案与用户评论区 -> 用户看到整体分布和高价值用户分布 -> 用户理解自己和高手判断差在哪 -> 用户私密保存/左滑保存至个人画像/关注问题/写评论。**

最优先验证的不是 AI 能不能生成内容，而是：

1. 这种选择题形式用户会不会连续刷；
2. 分布展示会不会增强印证感；
3. 高价值用户分布会不会带来认知差距和学习感；
4. 用户是否愿意把某些选择保存到个人画像；
5. 别人是否愿意打开主页看这个人的选择记录和评论；
6. 用户是否愿意先选择评论维度，再补一句短评论表达“我为什么选”；
7. 用户是否愿意关注问题，并回到关注副页面看更新；
8. 用户是否愿意设置特别关注，并看特别关注人的选项选择；
9. 用户是否愿意在主题区发帖，并让帖子沉淀到主页画像。

如果这些成立，这个产品就不只是“刷知识”，而是一个以选择数据、关注流、个人画像、评论维度、主题发帖和高价值用户分布为核心的数据飞轮。

## 七、遗待解决的风险

### 7.1 用户评论和平台标准答案如何融合

当前版本的原则是：**前期平台答案和用户评论完全分层，不自动融合。**

这样做可以避免几个早期风险：

- 用户评论质量不稳定，直接进入答案会污染平台解释；
- 平台答案如果频繁被评论改写，会削弱“平台答案”的稳定性；
- 如果由人工决定哪些评论进入答案，又会回到人力和品味裁判问题；
- 如果完全自动融合，又可能把少数高赞但偏激的评论包装成标准结论。

但长期看，这里仍然是一个未解决问题：**高质量用户评论到底如何反哺平台答案？**

后续可以继续思考三种方向：

| 方向 | 机制 | 风险 |
|---|---|---|
| 只作为参考 | 评论永远不进入平台答案，只给编辑和模型做选题参考 | 评论资产利用不足 |
| 半自动融合 | 系统周期性总结高赞/高质量评论，形成“社区补充观点” | 仍需处理噪声和偏见 |
| 版本化答案 | 平台答案保留主版本，用户评论触发“社区修订版/争议版” | 产品复杂度上升 |

这个问题不建议在 MVP 阶段解决。MVP 只验证平台答案与开放评论分层是否成立；等评论量、点赞、特别关注、高价值用户分布产生数据后，再判断是否需要融合。

## 八、产品调研补充附录

本节把产品调研报告中多出来、但需求分析正文没有完整展开的内容整理到最后，作为审阅时的补充判断。

### 8.1 外部机会判断

这个产品不是孤立出现的，它踩在三个已经被验证的趋势上：

| 趋势 | 已有参照 | 对本项目的启发 |
|---|---|---|
| 短知识消费 | Headway、Blinkist、短内容平台 | 用户愿意在碎片时间消费被压缩过的知识，但纯摘要容易变被动阅读 |
| 互动式学习 | Duolingo、Brilliant | “先做一个动作，再得到反馈”比单纯阅读更容易带来留存 |
| AI 生成学习材料 | NotebookLM、Quizlet | “材料 -> flashcards/quizzes”会成为基础能力，不能作为长期差异化 |

因此本项目的机会不在于“AI 能生成题”，而在于：

**把真实问题改造成用户愿意参与选择、愿意看分布、愿意看平台答案与评论区分歧的判断场。**

### 8.2 竞品格局与反定位

详细竞品分析单独沉淀在 [竞品分析](竞品分析/README.md)，本节只保留和需求判断直接相关的结论。

| 产品/类型 | 已经满足什么 | 留下的空位 |
|---|---|---|
| 知乎 / Quora | 高密度经验、问答、争论、案例 | 内容长、质量不稳定、筛选成本高 |
| 知乎直答 / AI Search | 综合回答、搜索式问答 | 用户需要先有问题，不适合无目的刷 |
| Duolingo | 短任务、反馈、留存、游戏化 | 主要是确定性学习，不覆盖真实决策判断 |
| Brilliant | 互动题、逐步理解、挑战感 | 更偏 STEM，不是职业/消费/生活经验判断 |
| Headway / Blinkist | 短知识、微学习、订阅 | 多数仍是读/听，参与感弱 |
| Quizlet / NotebookLM | 从材料生成 flashcards/quizzes | 更适合备考/材料学习，不适合无聊刷好问题 |
| Perplexity / ChatGPT Search | 即问即答、来源引用 | 是搜索入口，不是 feed 入口 |
| ELYS / AI 社交 | AI 分身、AI 评论制造互动感 | AI 不能替代真实他者确认，本项目不能伪造用户评论和高价值用户活跃 |

需要避开的反方向：

1. **不要做低配 Headway**：只把长内容压缩成摘要。
2. **不要做低配 Quizlet**：只把内容变成单选题。
3. **不要做 AI 伪互动社区**：AI 可以辅助整理和推荐，但不能伪造真人评论、点赞、保存或高价值用户分布。

更准确的定位是：

**多邻国式的知乎判断题，但不是学语言，而是训练用户对真实问题的判断力。**

### 8.3 产品命题抽象

本项目不应该被定义成“知乎内容摘要器”，而应该定义成：

**选择优先的知识判断训练平台。**

用户主动作不是“读”，而是：

```text
看到好问题 -> 做选择 -> 看平台答案 -> 看分布 -> 看评论区 -> 保存/关注/分享
```

这和普通知识卡片的区别是：

| 形态 | 用户动作 | 问题 |
|---|---|---|
| 摘要 App | 先读摘要 | 容易变被动消费 |
| Flashcard | 先记忆/回忆 | 更适合考试 |
| AI 搜索 | 先提问 | 需要明确需求 |
| 本产品 | 先选择，再看答案和分布 | 选择本身制造参与感和判断差异 |

### 8.4 题型系统

第一版不宜泛化成“所有知识题”，更适合先聚焦真实处境题：

| 题型 | 是否有标准答案 | 适配度 |
|---|---|---|
| 事实核验题 | 有 | 适合作为轻量补充，不做主线 |
| 概念辨析题 | 有较优解释 | 适合建立框架 |
| 处境取舍题 | 无唯一答案 | 最适合主 Feed |
| 坑点扫描题 | 有高概率关键变量 | 最容易制造获得感 |
| 预测判断题 | 概率型 | 适合商业、行业、AI 趋势 |
| 立场地图题 | 无正确答案 | 适合分享和评论 |

优先做：

**处境取舍题 + 坑点扫描题。**

这两类最能承接知乎类内容的价值：不是给一个唯一答案，而是让用户看到不同选择背后的变量、边界、案例和反例。

### 8.5 示例卡片

#### 示例 1：职业取舍

**题目**：第一份工作，你更应该优先选哪一个？

A. 大厂边缘岗  
B. 小公司核心岗  
C. 继续读研  
D. 先 gap 半年探索

平台答案不应该只说“B 更好”，而应该解释：大厂边缘岗给的是背书、流程和安全网；小公司核心岗给的是问题密度、反馈速度和真实业务接触；不同选择成立的条件不同。

#### 示例 2：AI 产品经理能力

**题目**：AI 产品经理最容易被高估的能力是什么？

A. 会写 Prompt  
B. 懂模型原理  
C. 会画流程图和 PRD  
D. 能设计评测和迭代闭环

题目最好问“最容易被高估”，而不是“最重要”，因为它更容易制造判断冲突，也更适合评论区分层。

#### 示例 3：消费决策

**题目**：判断一门课程值不值得买，最先看什么？

A. 老师履历  
B. 学员评价  
C. 课程大纲  
D. 是否能产出可验证结果

这类题适合解释“履历是信任入口，但不是学习结果；评价可能被筛选；大纲可能只是包装；真正关键是结果是否能被验证”。

### 8.6 内容生产 pipeline

第一版不要追求全自动生成。建议流程是：

```text
候选问题池
  -> 来源聚合
  -> 观点聚类
  -> 生成题干和选项
  -> 生成平台答案草稿
  -> 人工审核/改写
  -> 发布到主 Feed
  -> 根据选择分布、评论和关注数据反向优化
```

候选来源：

- 知乎高赞问答、热榜历史题、收藏夹；
- Reddit / Hacker News / Quora 高争议讨论；
- 播客文字稿；
- 书籍、报告、博客中的高密度观点；
- 自己整理的面试题、职业题、产品题、消费决策题。

AI 适合做：

- 从长文中抽取观点和变量；
- 把多个回答聚类成 3-4 个选项；
- 为每个选项生成成立条件、反例和边界；
- 生成不同难度和语气的题干；
- 帮编辑检查来源和解释是否空泛。

AI 不适合完全放手：

- 判断某个选项是否真的高价值；
- 判断争议题是否被过度简化；
- 判断来源是否可靠；
- 把一个案例包装成普遍规律。

### 8.7 增长机制补充

单卡钩子要让用户忍不住先选：

- “哪个选择看起来聪明，其实最危险？”
- “大多数人会选 A，但高质量用户更在意哪个变量？”
- “这件事真正的分水岭是什么？”
- “如果只能问一个问题，你会问什么？”

留存不只靠 streak，更应该靠判断资产：

- 本周完成 20 个职业判断；
- 解锁“求职坑点雷达”；
- 你的判断偏好：安全优先 / 增长优先 / 资源优先 / 自主性优先；
- 你最常忽略的变量：时间成本 / 机会成本 / 反馈密度 / 下行风险。

分享默认是图文卡片，不是长链接体系：

```text
问题 + 我的选择 + 选项分布 + 对应选项的平台答案
```

### 8.8 商业化和战略风险补充

短期不建议收费。先验证留存、评论区活性、关注问题回访和主页画像价值。

未来可能路径：

| 模式 | 适配度 | 说明 |
|---|---|---|
| 广告 | 中 | 要谨慎，避免污染决策信任 |
| 导流 | 中高 | 适合课程、书、工具、咨询，但必须和题卡强相关 |
| 品牌专题 | 中 | 可做职业避坑、AI 工具、消费决策专题 |
| 会员 | 后排 | 不作为当前主线 |
| 决策报告 | 高 | 基于用户选择记录生成判断偏好/盲区报告 |

主要战略风险：

- 变成低质冷知识；
- 变成考试刷题；
- 平台答案太像 AI 套话；
- 题目没有真实处境；
- 个性化过早；
- 内容供给拖慢迭代。

### 8.9 下一步建议

1. 先选两个垂直：职业/求职 + AI 产品/商业判断。
2. 人工做 100 张题卡，先不要工程化。
3. 用 Figma 或极简 Web 原型验证：选择、看答案、看分布、评论、保存、关注问题。
4. 每张卡记录：题干吸引力、作答率、评论率、关注率、私密保存率、个人画像保存率。
5. 等种子用户验证后，再做 AI 辅助生产和推荐机制。
