# 2026-07-10 v2 开发线：MVP 冻结，社交层五个功能进列表版

## 1. 当时的问题

MVP 在等领导审阅，main 分支直连 GitHub Pages，动一行线上演示就变。但功能不能停在原地：需求文档十个大功能，MVP 只覆盖了三个半（Feed、先选择后揭示、选项分布加分层雏形）。要在不碰审阅版的前提下把剩下的核心功能做出来，还得留一条干净的合并路。

## 2. 做出的判断

- 用 git worktree，不用复制目录。同一个仓库两个工作区：main 挂原目录纹丝不动，v2 分支挂新目录随便折腾，将来合并就是一条 git merge。复制目录看着省事，合并时是灾难。
- 功能只进列表版，滑版冻结。两版同步加等于每个功能的工作量翻倍，Isaiah 拍板先做列表版，验证完再谈滑版。
- 演示数据当产品资产做。126 条评论、60 位高手样本、40 条关注更新，按每张卡的选项和维度定制生成，过脚本校验：选项 key 合法、维度引用合法、零平台词汇。假数据也要像真的，但界面上明说是示意。
- 持久化边界照旧。作答不落盘，演示者每次打开都有题可答；保存、画像、关注落 localStorage，这些功能不持久化就不成立。两条规则服务的是同一件事：演示体验完整。

## 3. 关键修改

- 阶段0：index.html 拆出 assets/app.css 和 app.js。纯搬运，三个文件可逐字节拼回原文件。题卡数据仍内联，双击直开不破。
- 阶段1（功能七）：私密保存、存入画像（选择加一句判断理由）、canvas 图文分享卡、「我的」页三区。
- 阶段2（功能六）：评论区与平台答案分层，评论带同维度/新维度标注，三档筛选，发布仅本次会话可见。
- 阶段3（功能五）：分层分布条之上补「高手怎么选」，每题两三位职业身份样本，高价值/超高价值两档。
- 阶段4（功能八）：关注问题接真状态，顶部「关注」tab 成关注副页面（答案更新、热评、分布变化三类更新流），特别关注星标置顶。
- 阶段5（功能九）：「专题」tab 轻量主题讨论，自由发帖，不做审核流。
- 新数据文件走 data/*.js 挂全局变量。file:// 下 fetch 会被浏览器拦，这是「双击直开」约束下唯一顺手的数据加载姿势。

## 当日第二轮：评论区按反馈改三处（085bac0）

Isaiah 用了一轮，打回三条：

1. 发布入口埋在评论流底部——想写评论得先滑完所有评论，顺序反了。挪到评论列表上方。
2. 维度标注过度设计。每条评论顶着「同维度 · 故障率」「新维度 · 持有成本」，发布还要先从题卡维度里挑一个、新维度还得起名。砍到只剩同维度/新维度两个类型：发布时二选一，展示也只标这两个字。
3. 立场不显眼。「选 A」是右侧灰色小字，看评论根本注意不到立场。改成蓝底白字实心标签，紧跟用户名。

判断记录：具体维度名是信息，但在评论流的扫读场景里更是噪音——用户扫评论时要的是「这人选了什么、站哪边说话」，维度类型给个粗分类就够。数据不删，social.js 里的 dim 字段原样留着，砍的只是展示。立场是评论区的第一信息，视觉权重要对得上。

遗留问题：同维度/新维度该不该让用户自己标？发布时多一步选择就多一分流失，平台自动判断（评论语义对比题卡维度）可能才是终态——留给真实用户数据裁决。

## 当日第三轮：v2 从分支改成文件夹

Isaiah 的要求：别把 v2 直接以分支形式放 GitHub，仓库里单独开一个文件夹存 v2 相关的，之前的版本还在原位置直接维护。

判断记录：worktree + 分支对写代码的人是干净的合并路，对看 GitHub 页面的人是隐形的——切分支才能看到 v2，审阅的人和自己翻仓库都不方便。文件夹模型让两个版本在同一页面并排可见，Pages 顺带能直接跑 v2 演示。代价是 v2/ 里镜像了一份构建记录（老记录重复一份），为了文件夹自包含、站内链接不断，接受。

- main 上新增 `v2/` 文件夹 = v2 版 mvp + 构建记录的完整镜像；根目录审阅版文件零改动（只加不改，Pages 老 URL 内容不变）。
- 远端 v2 分支删除；本地 worktree（knowledge-choice-feed-v2/）保留作存档，后续开发直接在 `v2/` 文件夹里做。
- 合并路径同步改写：审阅通过后不再走 PR，直接用 v2/ 内容覆盖根目录。
- v2 线里的活构建记录是 `v2/构建记录/`，根目录那份停在审阅版状态不再动。
