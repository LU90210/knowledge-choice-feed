# 2026-07-09 买车决策 MVP：真实数据管线与知乎风演示

## 1. 这一步做了什么

产品从纯文档走到第一个可点的演示。范围只覆盖主链路：

```text
Feed 流 -> 问题卡片 -> 详情页 -> 做选择 -> Reveal（每个选项的理由与盲区）-> 分布图（整体 + 分层）
```

产出在 `mvp/` 下：

- `mvp/index.html`：单文件演示，手机端知乎的视觉语言，数据内嵌，file:// 直开。
- `mvp/data/cards.json`：20 张「买车决策」题卡，源数据真身。
- `mvp/data/raw/`：知乎数据开放平台的原始返回存档（12 次搜索 + 1 次热榜）。
- `mvp/pipeline/fetch_zhihu.sh`：拉取脚本，Access Secret 走环境变量，不落盘。

## 2. 数据是真的，分布是演示的

数据源用了知乎数据开放平台（developer.zhihu.com）的 `zhihu_search` 接口：12 个买车决策子题（奔驰还是宝马、SUV 还是轿车、油车还是电车、全款还是贷款……）各抓 10 条，共 120 条真实问题、回答与文章摘要。

题卡按需求文档的六字段 SOP 蒸馏：题目、领域、2-4 个选项、每个选项的成立理由、每个选项忽略的变量、选完后的「原来如此」。53 条来源链接经脚本核对，全部严格出自 API 原始返回，零杜撰。

选择分布和高价值用户分层是合成数据，界面上有标注。这条线不能糊：真实内容和演示数据要在产品里始终分得清，这是 ELYS 教训在数据层的版本。

## 3. 过程里长出来的判断

**题材从「AI 前沿技术」换成「买车决策」。** 换完才发现它更贴产品定位：买车是消费避坑主题里判断冲突最密集的场景。「没桩选混动还是纯电」下面，vote 104 和 vote 12 的两个回答结论完全相反，这种高赞互怼就是题卡的原材料，AI 话题里反而少见。

**搜索结果本身成了一道题。**「30 万落地买什么」抓回来的 10 篇里 6 篇是结构雷同的车型软文。这个发现直接写成了一张坑点扫描卡（q008）：先判断信息，再判断车。蒸馏管线不只提取观点，也顺手暴露了信息生态，这可能是产品的隐藏卖点。

**分布的叙事作用比预期大。** 分层分布在「这个价位不碰 BBA」这类选项上反差明显：普通用户 18%，超高价值用户 39%。不用解释，一眼就懂「懂的人选得不一样」。需求文档 2.6 的设计第一次在真题上看到了效果。

## 4. 还没做的

评论区、关注流、用户系统、左右滑手势、真实分布统计。需求文档里都有定义，这轮不做。
